# Creating a frontend module

OpenMRS publishes [`@openmrs/create-o3-app`](https://www.npmjs.com/package/@openmrs/create-o3-app), a generator for scaffolding O3 modules. It can generate standalone modules, modules inside existing monorepos, and new monorepos with the usual O3 conventions already wired up. It requires Node.js 20 or newer. The generator source lives in [`openmrs/create-o3-app`](https://github.com/openmrs/create-o3-app). The [Template](https://github.com/openmrs/openmrs-esm-template-app) frontend module is still useful as a stable reference or as a manual fallback.

<Callout type="warning">
  Do not run `npx create-o3-app`. The unscoped `create-o3-app` package on npm is an unrelated third-party project. The OpenMRS generator is published under the scoped name `@openmrs/create-o3-app`.
</Callout>

## Example: Creating a new frontend module

<Callout emoji="ℹ️" type="info">
  Note that we're setting this repo up as a
  [polyrepo](https://monorepo.tools/#what-is-a-monorepo). If you want to create
  a new frontend module in a monorepo, see the note about monorepos below.
</Callout>

Suppose you want to create a frontend module that provides a UI for displaying the active prescriptions. Our goal is to attempt to recreate the [`Active Prescriptions`](https://dev3.openmrs.org/openmrs/spa/dispensing) tab of the [`openmrs-esm-dispensing-app`](https://github.com/openmrs/openmrs-esm-dispensing-app).
The process for doing that would be as follows:

<Steps>

### Step 1

Run the published OpenMRS generator from the directory where you want the new module folder to be created. For this example, create a standalone module named `active-prescriptions`:

```sh
npm exec @openmrs/create-o3-app@latest -- active-prescriptions --standalone --package-name "@openmrs/esm-active-prescriptions-app" --route "/active-prescriptions" --route-component "ActivePrescriptions"
```

Rspack is the default build tool. Add `--webpack` only if you specifically need a Webpack scaffold. If you prefer to accept the defaults and answer any prompts that apply to your current directory, run `npm exec @openmrs/create-o3-app@latest -- active-prescriptions`.

<Callout emoji="ℹ️" type="info">
  Why the `esm` prefix?
  Frontend modules have a convention of being prefixed with `esm`. See the note about [frontend modules](/docs/frontend-modules/overview).
</Callout>

<Callout emoji="ℹ️" type="info">
  If you need to work from the Template repository directly, go to the [Template](https://github.com/openmrs/openmrs-esm-template-app) seed app, click `Use this template`, choose `Create a new repository`, and clone the new repository to your local machine.
</Callout>

### Step 2

You'll want to `cd` into your project directory. The CLI installs dependencies automatically unless installation is skipped, interrupted, or Corepack is unavailable. If you need to install manually, run:

```sh
cd openmrs-esm-active-prescriptions-app
corepack yarn install
```

For standalone modules, the generated directory is based on the package name. In this example, the `--package-name "@openmrs/esm-active-prescriptions-app"` flag creates the `openmrs-esm-active-prescriptions-app` directory.

Generated projects pin Yarn with the `packageManager` field, so use `corepack yarn ...` commands to respect the version in the generated `package.json`. If Corepack is not enabled on your machine, run `corepack enable` once and then rerun the install command.

Generated projects already depend on `openmrs@next`, `@openmrs/esm-framework@next`, and the selected OpenMRS build config. Use `corepack yarn up openmrs@next @openmrs/esm-framework@next @openmrs/rspack-config@next` (or `@openmrs/webpack-config@next` if you scaffolded with `--webpack`) only when you intentionally want to pull the newest prerelease core libraries into the project. For release work, pin tested versions instead of leaving them on moving tags.

Run `corepack yarn start` to get the app running. To run the app on a specific port, add the suffix `--port portNumber` to the start command.

The app should automatically open up in your browser, and you can use the below credentials to log in.
Username: `admin` 
Password: `Admin123`

Once logged in, you should be redirected to the home page of the dashboard. But, wait a second, what's the fancy home page with the sidebar and all the data, and where did the login page come from?
You'll notice that the code for the home page and the login page is not actually in your project directory. The [home](https://github.com/openmrs/openmrs-esm-patient-management/tree/main/packages/esm-home-app) and [login](https://github.com/openmrs/openmrs-esm-core/tree/main/packages/apps/esm-login-app) modules live in separate packages.

The data is fetched from the hosted dev server. If you want to connect it to a different backend, append the suffix `--backend https://your-backend-url` to the `corepack yarn start` command.

TL;DR: the `corepack yarn start` command actually runs `openmrs develop`, which sets up the local app shell according to its configuration. Read more about [configuration](/docs/configuration-system).

<Callout emoji="ℹ️" type="info">
  To learn more about how `openmrs develop` works, check out [`runDevelop`](https://github.com/openmrs/openmrs-esm-core/blob/main/packages/tooling/openmrs/src/commands/develop.ts). Note that the Template repository is already set up for Core v5+ and uses `routes.json` and `startupApp()` for module registration. If you're migrating an older module, see the [migration guide](/docs/migrate-to-core-v5).
</Callout>

### Step 3

Next, review the generated files and adapt them to your needs. If you used `@openmrs/create-o3-app`, most of these values should already match the flags or prompts you gave the CLI. If you used the Template directly, you'll need to rename the template values yourself. Check the following:

- Replace any remaining instances of `template` with `active-prescriptions`.
- Confirm the module metadata:

  - `src/constants.ts` should export `moduleName = "@openmrs/esm-active-prescriptions-app"`.
  - `src/index.ts` should pass that `moduleName` into the lifecycle options.
  - If the generated entry point still has a placeholder `featureName`, change it to a more useful value for this module.
  - Ensure you have a `startupApp()` function that calls `defineConfigSchema()` if your module has configuration. For Core v5+ modules, this is required for configuration schemas to be registered properly.

- Update the route to the frontend module.

  - We need a route URL to access the frontend module. This is specified in `src/routes.json` in the `pages` property. `create-o3-app` registers the generated route against the exported `root` lifecycle, and `root.component.tsx` renders the route component.

  ```json filename="src/routes.json"
  {
    "$schema": "https://json.openmrs.org/routes.schema.json",
    "pages": [
      {
        "route": "active-prescriptions",
        "component": "root",
        "online": true,
        "offline": true
      }
    ],
    "extensions": []
  }
  ```

Now that the route is set up, it can be accessed via `http://localhost:8080/openmrs/spa/active-prescriptions`, which should show the generated boilerplate for the new app.
The boilerplate has useful resources about key features of the O3 framework that will likely help you in the development of your frontend module. To understand the underlying template, you can read the `README.md` of the [template repository](https://github.com/openmrs/openmrs-esm-template-app/blob/main/README.md).

### Step 4

Profit! You should now have a custom frontend module that you can use in your OpenMRS application.

</Steps>

### Creating a frontend module in a monorepo

The process for creating a new frontend module in a monorepo is nearly identical to the one for creating one in a polyrepo with a few tweaks:

1. From the monorepo root, run `npm exec @openmrs/create-o3-app@latest -- referrals-queue --monorepo` and answer the prompts. In non-interactive mode, the CLI defaults to `packages/apps/esm-referrals-queue`. To create a new monorepo root instead, use `--new-monorepo`.
2. If you copy the Template manually instead, put its contents into a new directory under `packages/`. For our example above, we'd create a new directory called `packages/esm-referrals-queue` and move the contents of Template into it.
3. Remove configuration files that are already present at the workspace level. These include files and folders such as `.github` and `.eslintrc`.
4. Remove `devDependencies` from the new frontend module's `package.json`
   which are already in the `devDependencies` of the `package.json` at
   the workspace level.

That's it!

To learn more about what's possible with frontend modules, read the [Overview](/docs/frontend-modules/overview) guide.

## Integrating your frontend module into your application

Once you've created a frontend module, you'll want to integrate it into your application. There are two steps to doing so.

First, publish your frontend module package to npm. At the end, your package should be visible on npmjs.com,
such as [`@openmrs/esm-login-app`](https://www.npmjs.com/package/@openmrs/esm-login-app).

Then, you'll need to integrate this package into your distribution. Add your module to the `frontendModules` object in your distro's `spa-assemble-config.json` file. For more information, see the [Creating a distribution](/docs/recipes/create-a-distribution) guide and the [Configuration overview](/docs/configure-o3/overview#spa-assemble-configjson) guide.
