# Création d'un module frontal

OpenMRS dispose d'un générateur dans [`openmrs/create-o3-app`](https://github.com/openmrs/create-o3-app) pour créer des modules O3. Il peut créer des modules autonomes, des modules dans un monorepo existant et de nouveaux monorepos avec les conventions O3 habituelles déjà configurées. Il nécessite Node.js 20 ou plus récent. Le module frontal [Template](https://github.com/openmrs/openmrs-esm-template-app) reste utile comme référence stable ou comme solution manuelle.

<Callout type="warning">
  Ne lancez pas `npx create-o3-app`. Le package `create-o3-app` non scopé sur npm est un projet tiers sans rapport. Le générateur OpenMRS est publié sous le nom scopé `@openmrs/create-o3-app`. Tant que ce package n'est pas disponible sur npm, construisez le générateur depuis les sources comme ci-dessous.
</Callout>

## Exemple : Création d'un nouveau module frontal

<Callout emoji="ℹ️" type="info">
  Notez que nous configurons ce dépôt en tant que
  [polyrepo](https://monorepo.tools/#what-is-a-monorepo). Si vous souhaitez créer
  un nouveau module frontal dans un monorepo, consultez la note sur les monorepos ci-dessous.
</Callout>

Supposons que vous souhaitiez créer un module frontal qui fournit une interface utilisateur pour afficher les prescriptions actives. Notre objectif est de tenter de recréer l'onglet [`Active Prescriptions`](https://dev3.openmrs.org/openmrs/spa/dispensing) de l'application [`openmrs-esm-dispensing-app`](https://github.com/openmrs/openmrs-esm-dispensing-app).
Le processus pour cela serait le suivant :

<Steps>

### Étape 1

Clonez et construisez le générateur OpenMRS:

```sh
git clone https://github.com/openmrs/create-o3-app.git
cd create-o3-app
npm install
npm run build
cd ..
```

Lancez ensuite la commande de génération depuis le répertoire où le nouveau dossier de module doit être créé. Pour cet exemple, créez un module autonome nommé `active-prescriptions`:

```sh
node create-o3-app/dist/index.js active-prescriptions --standalone --package-name "@openmrs/esm-active-prescriptions-app" --route "/active-prescriptions" --route-component "ActivePrescriptions"
```

Si vous préférez accepter les valeurs par défaut et répondre aux prompts qui s'appliquent au répertoire courant, lancez `node create-o3-app/dist/index.js active-prescriptions`.

<Callout emoji="ℹ️" type="info">
  Pourquoi le préfixe `esm`?
  Les modules frontaux ont la convention d'être préfixés par `esm`. Voir la note sur [les modules frontaux](/docs/frontend-modules/overview).
</Callout>

<Callout emoji="ℹ️" type="info">
  Si vous devez partir directement du dépôt Template, ouvrez l'application seed [Template](https://github.com/openmrs/openmrs-esm-template-app), cliquez sur `Use this template`, choisissez `Create a new repository`, puis clonez le nouveau dépôt sur votre machine locale.
</Callout>

### Étape 2

Entrez dans votre répertoire de projet et installez les dépendances si le CLI ne l'a pas déjà fait:

```sh
cd openmrs-esm-active-prescriptions-app
corepack yarn install
```

Les projets générés épinglent Yarn avec le champ `packageManager`, donc utilisez les commandes `corepack yarn ...` pour respecter la version indiquée dans le `package.json` généré. Si Corepack n'est pas activé sur votre machine, lancez `corepack enable` une fois, puis relancez l'installation.

Les projets générés dépendent déjà de `openmrs@next`, `@openmrs/esm-framework@next` et de la configuration de build OpenMRS sélectionnée. Utilisez `corepack yarn up openmrs@next @openmrs/esm-framework@next` uniquement si vous voulez volontairement récupérer les bibliothèques core prerelease les plus récentes. Pour un travail de release, épinglez plutôt des versions testées.

Exécutez `corepack yarn start` pour lancer l'application. Pour exécuter l'application sur un port spécifique, vous pouvez ajouter le suffixe `--port portNumber` à la commande start.

L'application devrait s'ouvrir automatiquement dans votre navigateur, et vous pouvez utiliser les identifiants ci-dessous pour vous connecter.

Nom d'utilisateur : `admin` 
Mot de passe : `Admin123`

Une fois connecté, vous devriez être redirigé vers la page d'accueil du tableau de bord. Mais, attendez une seconde, d'où vient cette page d'accueil sophistiquée avec la barre latérale et toutes les données, et où est passée la page de connexion ?
Vous remarquerez que le code de la page d'accueil et de la page de connexion n'est en fait pas dans votre répertoire de projet. Les [home](https://github.com/openmrs/openmrs-esm-home) et [login](https://github.com/openmrs/openmrs-esm-core/tree/main/packages/apps/esm-login-app) sont des applications à part entière.

Les données sont récupérées à partir du serveur de développement hébergé. Si vous souhaitez le connecter à un autre backend, vous pouvez ajouter le suffixe `--backend https://your-backend-url` à la commande `corepack yarn start`.

TL;DR : la commande `corepack yarn start` exécute en fait `openmrs develop` qui, selon une certaine configuration, configure le tout (lisez-en plus sur la [configuration](/docs/configuration-system)).

<Callout emoji="ℹ️" type="info">
  Pour en savoir plus sur le fonctionnement de `openmrs develop`, consultez [`runDevelop`](https://github.com/openmrs/openmrs-esm-core/blob/main/packages/tooling/openmrs/src/commands/develop.ts). Notez que le dépôt Template est déjà configuré pour Core v5+ et utilise `routes.json` et `startupApp()` pour l'enregistrement des modules. Si vous migrez un module plus ancien, consultez le [guide de migration](/docs/migrate-to-core-v5).
</Callout>

### Étape 3

Ensuite, vérifiez les fichiers générés et adaptez-les à vos besoins. Si vous avez utilisé `create-o3-app`, la plupart de ces valeurs devraient déjà correspondre aux options ou aux réponses données au CLI. Si vous avez utilisé le Template directement, vous devrez renommer les valeurs du template vous-même. Vérifiez les points suivants:

- Remplacer toutes les instances de `template` par `active-prescriptions`.
- Vérifier les métadonnées du module:

  - `src/constants.ts` doit exporter `moduleName = "@openmrs/esm-active-prescriptions-app"`.
  - `src/index.ts` doit passer ce `moduleName` dans les options de lifecycle.
  - Si le point d'entrée généré contient encore un `featureName` de placeholder, remplacez-le par une valeur plus utile pour ce module.
  - Assurez-vous d'avoir une fonction `startupApp()` qui appelle `defineConfigSchema()` si votre module a une configuration. Pour les modules Core v5+, cela est requis pour que les schémas de configuration soient correctement enregistrés.

- Mettre à jour la route vers le module frontal.

  - Nous avons besoin d'une URL de route pour accéder au module frontal. Celle-ci est spécifiée dans `src/routes.json` dans la propriété `pages`. `create-o3-app` enregistre la route générée contre le lifecycle exporté `root`, et `root.component.tsx` rend le composant de route.

  ```json filename="src/routes.json"
  {
    "$schema": "https://json.openmrs.org/routes.schema.json",
    "pages": [
      {
        "route": "active-prescriptions",
        "component": "root",
        "online": true,
        "offline": true
      }
    ],
    "extensions": []
  }
  ```

Maintenant que la route est configurée, elle peut être accessible via `http://localhost:8080/openmrs/spa/active-prescriptions`, ce qui devrait afficher le boilerplate généré pour la nouvelle app.
Le boilerplate contient des ressources utiles sur les fonctionnalités clés du framework O3 qui vous aideront probablement dans le développement de votre module frontal. Pour comprendre le template sous-jacent, vous pouvez lire le `README.md` du [dépôt template](https://github.com/openmrs/openmrs-esm-template-app/blob/main/README.md).

### Étape 4

Profitez ! Vous devriez maintenant avoir un module frontal personnalisé que vous pouvez utiliser dans votre application OpenMRS.

</Steps>

### Création d'un module frontal dans un monorepo

Le processus de création d'un nouveau module frontal dans un monorepo est presque identique à celui de la création dans un polyrepo, avec quelques ajustements :

1. Depuis la racine du monorepo, lancez `node /path/to/create-o3-app/dist/index.js referrals-queue --monorepo` et répondez aux prompts. En mode non interactif, le CLI utilise `packages/apps/esm-referrals-queue` par défaut.
2. Si vous copiez le Template manuellement, placez son contenu dans un nouveau répertoire sous `packages/`. Pour notre exemple ci-dessus, nous créerions un nouveau répertoire appelé `packages/esm-referrals-queue` et déplacerions le contenu du Template dedans.
3. Supprimez les fichiers de configuration déjà présents au niveau de l'espace de travail. Il s'agit de fichiers et de dossiers tels que `.github` et `.eslintrc`.
4. Supprimez les `devDependencies` du nouveau module frontal dans le `package.json`
   qui sont déjà dans les `devDependencies` du `package.json` au niveau de l'espace de travail.

C'est tout !

Pour en savoir plus sur les possibilités des modules frontaux, consultez le guide [Aperçu](/docs/frontend-modules/overview).

## Intégration de votre module frontal dans votre application

Une fois que vous avez créé un module frontal, vous voudrez l'intégrer dans votre application. Il y a deux étapes à suivre pour cela.

Tout d'abord, publiez votre package de module frontal sur NPM. À la fin, votre package devrait être visible sur npm.js,
comme [`@openmrs/esm-login-app`](https://www.npmjs.com/package/@openmrs/esm-login-app).

Ensuite, vous devrez intégrer ce package dans votre distribution. Ajoutez votre module à l'objet `frontendModules` dans le fichier `spa-assemble-config.json` de votre distribution. Pour plus d'informations, consultez le guide [Créer une distribution](/docs/recipes/create-a-distribution) et le guide [Vue d'ensemble de la configuration](/docs/configure-o3/overview#spa-assemble-configjson).
