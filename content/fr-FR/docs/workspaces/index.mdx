# Vue d'ensemble

Les espaces de travail sont le modèle d'interaction latéral d'O3 pour la saisie de données, les commandes, le remplissage de formulaires et les autres tâches ciblées. Sur ordinateur, les fenêtres d'espace de travail s'ouvrent sur le côté de l'écran. Sur tablette et téléphone, elles s'affichent en superposition.

Workspace v2 utilise un modèle à trois niveaux :

- **Groupe d'espaces de travail** - Le conteneur principal d'un flux de travail lié. Un groupe peut posséder un menu d'actions, définir où il persiste avec `scopePattern`, et choisir s'il s'affiche en superposition.
- **Fenêtre d'espace de travail** - Un panneau dans un groupe. Les fenêtres définissent la largeur, le comportement d'agrandissement, l'ordre d'affichage et éventuellement une icône de menu d'actions.
- **Espace de travail** - Le composant React qui rend l'interface de tâche dans une fenêtre. Une fenêtre peut contenir une pile d'espaces de travail, ce qui représente les flux parent et enfant.

Un composant d'espace de travail est enregistré dans `routes.json` sous `workspaces2`. Chaque espace de travail pointe vers une entrée `workspaceWindows2`, et chaque fenêtre pointe vers une entrée `workspaceGroups2`. Les applications peuvent définir leur propre groupe ou rattacher des fenêtres à un groupe existant comme `patient-chart` ou `ward-patient`.

Les composants d'espace de travail doivent rendre `<Workspace2>` comme conteneur principal. Lancez les espaces de travail avec `launchWorkspace2`, ou utilisez des helpers patient comme `useLaunchWorkspaceRequiringVisit` quand un flux nécessite une visite active.

Les anciennes APIs v1 (`workspaces`, `launchWorkspace`, `ActionMenuButton`, le `type` d'espace de travail, `canHide` et `promptBeforeClosing`) sont dépréciées. Consultez [Migrer vers Workspace v2](/docs/migrate-to-workspace-v2) lorsque vous mettez à jour du code plus ancien.
